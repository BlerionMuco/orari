# Booking SaaS — V1 Deliverables

**V1 goal:** one real business can sign up, set itself up, and take live bookings end to end. Ship the spine to a single Tirana shop and let their usage tell you what to build next.

**V1 ground rules (locked):**
- No customer payments — money changes hands in person (no deposits, no payment step, no PCI scope).
- Customers are accountless — book and manage via a tokenized link.
- Multi-barber from day one — a business can invite its team via email (`add_team_member` RPC) and edit per-resource working hours. The Resource abstraction (`staff | asset`) carries this. (Originally "one resource per business to start" — relaxed because the team + per-resource hours editor shipped in dashboard M3 and unblocks real Tirana shops, which all have ≥2 barbers.)
- No-show defence = reminders + frictionless reschedule + no-show history (not deposits).
- Free trial, clock starts at go-live, no card up front.

## Part A — UI deliverables (screens & components)

### Marketing / onboarding surface (SSR, Calm Neutral)
- **Landing / marketing page** — *late V1, final pre-go-live task* (see sequencing note below)
- Business sign-up screen + email-verification state
- Login, password-reset request, password-reset confirm
- Onboarding wizard shell with step progress, plus five step screens:
  business profile (+ vertical) → services → working hours → team (owner + email invites) → go-live (slug w/ live availability check, public URL shown)
- Invite-accept page (`/invite/[token]`) — server resolves token kind (not_found / expired / accepted / revoked / pending); pending → accept card; everything else → ScreenState. Wraps the `accept_invite` SECURITY DEFINER RPC.

### Public booking surface (SSR, mobile-first, channel theme — Warm Boutique or Crisp Court)
- Business public page (profile header, services list)
- Service selection
- Resource selection (skippable when there's only one)
- Time picker — the centerpiece: real availability, available / selected / unavailable slot states, explicit timezone label
- Guest details form (name, phone, email — no account)
- Confirmation screen
- Manage-booking page via tokenized link (view / cancel / reschedule)
- 404 + "business not found" + "no availability" states

### Dashboard surface (Calm Neutral, mobile-first to 375px; desktop sidebar + mobile bottom-tab shell)
- Overview / home — greeting, Next booking card, Later today list; role-aware (staff filtered to own resource)
- Day view of bookings (read-only list for V1 — not the drag calendar); day strip + resource filter chips (owner)
- Booking detail with status actions (complete / no-show / cancel); contact (tel/mailto), notes, no-show history flag (`countNoShowsForCustomer`)
- Manual booking + block-time (`/dashboard/new-booking?mode=booking|block`) — segmented mode tabs (staff = block only, server-enforced)
- Services management (CRUD) — name, duration, price, before/after buffers, active toggle, delete (confirm)
- Team & invites — members list w/ login pill + active toggle; pending invites (resend / revoke); invite sheet wraps `add_team_member` RPC
- Working-hours editor — owner resource picker; 7-day rows w/ open switch + time-select; "apply first-open day to all"
- Booking-rules settings — lead time (hours), advance window (days), slot granularity (min) on the business-default row
- Reminders settings — `reminder_enabled` + `reminder_offsets_min[]` (off / 24h / 1h / both); dispatcher to follow
- Business profile / public-page basics editor — name, slug (live availability check), phone, description, location, currency, copy public URL
- Account / settings shell — settings menu (role-aware: Shop section owner-only, You section everyone); account form (name + email + change password via Supabase `updateUser`); sign-out card w/ destructive confirm
- Billing (placeholder) — trial-status card (status pill + days left + ends date), included-features stub, "coming soon" upgrade CTA; reads the subscription scaffold (no write surface — Paddle/Lemon Squeezy out of V1)
- Restricted screen for staff-blocked routes (`/dashboard/restricted`)

### Cross-cutting UI
- Shared primitives in `src/components/ui/<category>/` — buttons, form, feedback (Toast / ScreenState / Skeleton / StatusBadge), overlay (ConfirmDialog / Sheet / Dialog), display (SectionCard / SettingRow / ListCard), date (DayStrip / DatePill), nav (NavLink / PageHeader)
- Empty / loading / error / restricted states via `ScreenState` reused across every dashboard list / detail / settings view
- Toast / confirm dialogs mounted once in the dashboard layout; driven by Zustand stores (`use-toast`, `use-confirm`); danger confirm = solid-red bg
- Mobile layouts for all public + dashboard screens (375px primary surface; desktop locks `h-dvh` and scrolls inside `<main>`, mobile uses body scroll w/ fixed bottom tabs)
- Accessibility pass (focus, labels, keyboard) on the Radix-backed components *(pending)*
- Motion set: slide between booking steps, slot select, success check *(pending)*

## Part B — Feature deliverables (what must actually work)

### Foundation
- Project scaffold (Next.js App Router, Tailwind, Radix, RHF/Zod, Zustand, React Query, TypeScript) with the four route groups: `(auth)`, `(public)`, `(dashboard)`, plus `/invite/[token]` under `(auth)`
- Schema + migrations (Drizzle): `business_id` on every table, RLS on, and the Postgres **exclusion constraint** on overlapping bookings; `booking_services` junction (0009); onboarding RPC `create_business_onboarding` (0010); subscription scaffold `subscription_status` enum + `trial_ends_at` + reminder settings columns (0011); `trial_ends_at` default + backfill (0012)
- Auth + multi-tenancy: signup, email verify, login, reset — every query scoped to `business_id` via `businessScope()` (Drizzle bypasses RLS so the scoped WHERE is the tenant boundary; RLS stays as a backstop)
- SECURITY DEFINER RPCs: `create_business` / `add_team_member` / `accept_invite` / `create_business_onboarding`

### The core engine (build slowly, test hardest)
- `getAvailableSlots()` — working hours, time-off, buffers, lead time, advance window, candidate generation, rule filtering
- Booking-creation transaction — re-validate + rely on the exclusion constraint to win double-booking races
- Timezone correctness — UTC storage, business-timezone math, DST boundaries

### Booking loop
- Resource abstraction with `staff | asset` type from day one
- Onboarding that produces a live, shareable public URL
- Guest booking (tokenized manage link, no account)
- Cancel + reschedule via that link *(BE wired — `loadBookingByManageToken` + `cancelBooking`; public manage UI still pending)*
- Confirmation email + a scheduled reminder (Resend/Postmark + Inngest) *(reminder offsets now configurable per business via 0011; dispatcher wiring pending)*

### Dashboard operations
- **Manual booking + block-time** — *belongs in V1, not v2* — keeps the product correct when a barber takes a walk-in or a phone booking. Shipped under `/dashboard/new-booking` (segmented mode tabs; block writes a `time_off` row that disappears from public availability)
- Status transitions (complete / no-show / cancelled) and basic no-show history (`countNoShowsForCustomer` surfaces a flag on the detail page)
- Services / resource / working-hours / rules / reminders editing wired to the engine
- Team & invites — owner can invite barbers via email (`add_team_member` RPC); pending invites can be resent or revoked; resources can be toggled active

### Launch plumbing
- Subscription scaffold only: `subscription_status` enum (trial / active / past_due / cancelled) + `trial_ends_at` + Billing read surface (`lib/billing/queries.ts::getSubscriptionState`); entitlements gating stub — real merchant-of-record billing can land before the first trial expires
- Rate limiting on the public unauthenticated surface *(pending)*
- Basic product analytics on the booking funnel (to earn the 20-second target) *(pending)*
- Transactional email deliverability set up *(pending)*

## Landing page sequencing

The marketing landing page is the **final pre-go-live task** — built late in V1, ready before launch, blocking none of the spine.

- It is **not an acquisition channel** at launch — first shops come from in-person sales in Tirana.
- Its job at go-live is **credibility**: a barber you're pitching may Google the name; a real page makes you look like a company, not a side project, which matters in a small word-of-mouth market.
- Keep it small: look legitimate, state the no-show value plainly, offer a sign-up path. **No** SEO depth, multiple variants, conversion funnels, or blog. Build the credible one-pager and move on.

Until it's built, the sign-up route just needs to be reachable directly (a `/signup` link, or a near-bare root pointing to it).

## Explicitly NOT in V1 (the scope guard)

Letting these creep in is the single most likely failure mode. All correct *eventually*; none earns its place until one real shop is running on the spine.

Drag calendar · customer accounts · public-page editor · analytics dashboards · deposits / customer payments · real billing flows · the marketplace.

---

## Two things to weight heaviest tomorrow

1. **The availability engine + exclusion constraint** is the real product; everything else is assembly. Spend disproportionate time here — timezones, DST, buffers, and the DB-level race fix are where booking apps quietly break.
2. **Manual booking / block-time** is easy to mentally file under "dashboard depth" and defer. Don't — without it the product breaks the first time a shop takes a walk-in, which is day one.
